# 기본 문의함 도입 계획

## 1. 개요
- 리브랜딩 전환 공지보다 먼저, 사용자가 ottline의 기본 사용 중 질문과 불편을 남길 수 있는 최소 창구를 연다.
- 첫 단계는 "브랜드 선호 조사"가 아니라 "기본 사용 지원 및 제품 개선 신호 수집"에 집중한다.
- 현재는 외부 폼이 아니라 서비스 내부 문의 스레드와 관리자 답변 화면까지 포함한 1차 구현이 들어간 상태다.

## 2. 우선순위
- 분류: P1
- 이유:
  - 삭제 동기화/복구 같은 핵심 데이터 안정성(P0)보다 후순위다.
  - 다만 리브랜딩 이전에 사용자 질문 수집 경로를 확보해 전환 리스크를 낮출 가치가 있다.

## 3. 목표 / 비목표

### 목표
1. 사용자가 "질문", "버그 제보", "기능 제안"을 보낼 수 있는 단일 진입점을 연다.
2. 앱 안에서 2번 이내 탭으로 접근 가능하게 만든다.
3. 한글/영문 사용자 모두 이해 가능한 짧은 안내 문구와 CTA를 제공한다.
4. 사용자는 자기 문의만 보고, 관리자는 전체 문의를 보고 답변할 수 있게 한다.

### 비목표
1. 앱 내 실시간 고객센터나 채팅 시스템을 만들지 않는다.
2. 실시간 푸시 알림이나 운영자 알림 채널까지 이번 단계에서 필수 구현으로 묶지 않는다.
3. 리브랜딩 설문, NPS, 장문 리서치 폼까지 확장하지 않는다.

## 4. 현재 기준 영향 범위
- API: 있음
- DB/스키마: 있음
- Sync/Local-first: 없음
- UI: 있음
- 다국어(i18n): 있음
- 운영 알림: 후속 검토

현재 구현은 백엔드 API + DB + 프론트 UI까지 포함한다.

## 5. 1차 구현 결정

### 결정
- 1차 릴리스는 자체 API 기반 내부 문의함으로 운영한다.

### 이유
1. 사용자와 관리자 모두 앱 안에서 문의 흐름을 끝낼 수 있다.
2. 사용자별 접근 제어가 가능하다.
3. 이후 알림, 필터, 운영 지표를 붙이기 쉽다.

## 6. 사용자 진입점

### 우선 진입점
1. 설정 화면 `apps/web/app/[locale]/account/page.tsx`
   - 가장 명시적인 진입점
   - "내 문의함 열기" 섹션 추가
2. 푸터 `apps/web/components/AppFooter.tsx`
   - 안내 그룹에 문의 링크 포함

### 후순위 진입점
1. FAQ 페이지
   - "찾는 답이 없나요?" 다음 액션으로 연결
2. 빈 상태 화면
   - 기록이 없거나 검색 결과가 없을 때 가벼운 문의 링크 제공 가능
3. 오류/오프라인 화면
   - 즉시 문의보다는 FAQ 또는 나중에 문의하도록 유도하는 수준으로 검토

## 7. UX 원칙
1. CTA는 "문의하기"와 "피드백 보내기" 중 하나로 통일한다.
2. 사용자는 "무엇을 적어야 하는지" 바로 이해할 수 있어야 한다.
3. 사용자는 앱 내부에서 문의를 작성하고, 답변도 앱 내부에서 확인한다.
4. 리브랜딩 맥락은 전면에 내세우지 않는다.
5. 질문 제출보다 "안심하고 물어볼 수 있다"는 인상을 먼저 만든다.

## 8. 현재 구현 상태

### 사용자 화면
1. `apps/web/app/[locale]/feedback/page.tsx`
   - 새 문의 작성
   - 내 문의 목록 확인
   - 관리자 답변 확인
2. 카테고리
   - `QUESTION`, `BUG`, `IDEA`, `OTHER`
3. 본인 문의만 조회 가능

### 관리자 화면
1. `apps/web/app/[locale]/admin/feedback/page.tsx?token=...`
2. 전체 문의 목록 조회
3. 스레드 상세 조회
4. 관리자 답변 등록

### 접근 제어
1. 사용자 API는 `X-User-Id` 기준으로 본인 문의만 반환
2. 관리자 API는 `X-Admin-Token` 기준으로 전체 문의 접근 허용
3. unlink된 기기는 `X-User-Id` + `X-Device-Id` 검증에서 차단
4. 차단 응답을 받은 클라이언트는 로컬 캐시와 계정 상태를 초기화

## 10. 현재 운영 알림 상태

1. 신규 문의 등록 시 Telegram 알림 전송 코드 추가
2. 알림은 환경변수 설정 시에만 활성화
3. 관리자 답변 등록 시 알림은 아직 전송하지 않음

### Telegram 설정
1. `TELEGRAM_NOTIFY_ENABLED=true`
2. `TELEGRAM_BOT_TOKEN`
3. `TELEGRAM_CHAT_ID`
4. 선택: `TELEGRAM_SERVICE_NAME`

### 현재 메시지 내용
1. 서비스명
2. 문의 카테고리
3. 제목
4. user 식별자 일부
5. 본문 미리보기

## 11. 현재 한계

1. 관리자는 여전히 관리자 페이지에서 실제 답변 작업을 해야 한다.
2. 관리자 목록 필터/검색/미답변 카운트는 아직 없다.
3. 사용자 추가 댓글(후속 메시지) 기능은 아직 없다.
4. Telegram 전송 성공/실패를 관리자 UI에서 확인할 수는 없다.

## 12. 검증 시나리오
1. 사용자가 문의를 등록하면 본인 문의함 목록/상세에 즉시 보인다.
2. 다른 사용자로 접속하면 본인 문의만 보이고 남의 문의는 보이지 않는다.
3. 관리자는 전체 문의 목록과 상세를 볼 수 있다.
4. 관리자가 답변하면 사용자 문의 상세에 답변이 표시된다.
5. 모바일에서 설정 화면의 문의함 버튼과 문의 작성/목록 UI가 깨지지 않는다.
6. Telegram 설정이 유효하면 신규 문의 등록 시 운영 채널에 알림이 온다.
7. Telegram 전송이 실패해도 문의 저장 자체는 실패하지 않는다.
8. unlink된 기기에서 타임라인/문의함/설정 진입 시 서버 접근이 차단되고 로컬 상태가 초기화된다.
